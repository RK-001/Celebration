## Part 1 — Architecture Decisions (Finalized)

### Decision 1: Keep GitHub Copilot SDK ✅

**Confirmed.** Organization trusts Microsoft/GitHub for data sovereignty — no data leaves the boundary. Billing is not a concern. The SDK is backed by GitHub, actively developed (v0.1.32+, 29+ releases), and the BYOK escape hatch exists if ever needed. The `copilot` CLI binary dependency is an operational concern to address in deployment packaging (Docker image must include it).

---

### Decision 2: No pgvector / Vector DB — Current KB Approach is Right ✅

**Confirmed and analytically validated.** The `fetch_kb_file` tool + INDEX.md navigation + LRU cache is the correct approach for this use case.

**Why it works:**

The typical UD conversation context budget:
```
System prompt + INDEX.md + README.md:     ~8-12K tokens
User messages (3-4 rounds):               ~2-4K tokens
Assistant messages (questions + draft):   ~8-15K tokens
Fetched KB files (6-8 files × ~4K avg):  ~24-32K tokens
─────────────────────────────────────────────────────
Total:                                    ~42-63K tokens
```
Most Copilot models have 128K-200K context windows. Peak usage is ~30-50% of window. `infiniteSessions` handles compaction if ever needed. There is no context pressure problem.

**Why pgvector is premature:**
- GitHub Agentic Workflows auto-generate `INDEX.md` and supporting docs on every PR merge to main → the primary failure mode of INDEX-based navigation (stale index) is already solved
- A typical requirement touches 2-3 existing modules → 2-3 KB files → 8-10 files across a full conversation at most
- pgvector would add: PostgreSQL dependency, embedding pipeline per KB update, chunking strategy decisions, embedding model costs, and new failure modes — for no measurable benefit at current scale

**Revisit pgvector only if:** KB exceeds ~150 files OR users consistently report "the AI didn't find relevant info." PostgreSQL will already be in place by then (Phase 1), making pgvector a low-cost add-on at that point.

**Two small improvements to make instead (low effort, high value):**

1. **Add size guard in `fetch_kb_file`** — currently `knowledgeBase.ts` has no per-file size limit. If a KB file exceeds ~30K chars it goes straight into context unchecked. Add truncation with a note (same pattern as `copilot.ts` summarizer):

   ```typescript
   // In knowledgeBase.ts → fetchKBFileContent(), after githubFetch()
   const MAX_KB_FILE_CHARS = 30_000;
   if (content.length > MAX_KB_FILE_CHARS) {
     content = content.slice(0, MAX_KB_FILE_CHARS) +
       "\n\n⚠️ [File truncated at 30,000 characters]";
   }
   ```

2. **Make INDEX.md entries descriptive** — the LLM decides which files to fetch based on how well INDEX entries describe content. Auto-generated INDEX should include 1-line descriptions per file, not just file paths:
   ```markdown
   ## Good
   - `payments.md` — PaymentService, refund logic, retry policies, webhook handlers
   
   ## Not enough
   - `payments.md`
   ```
   Ensure the Agentic Workflow prompt that generates INDEX.md produces descriptive entries.

### Decision 3: Multi-Project KB Mapping ✅

Users select which project they want to work on → backend maps project name to its KB repo from a stored config. All KB repo configurations stored in a `projects` database table rather than env vars. This supports multiple teams/projects from a single deployment.

---

### Decision 4: Session Management Improvements ✅

- **Redis for session metadata** — SDK sessions stay in-process (not serializable), Redis tracks ownership per server instance for sticky routing in multi-instance deployments
- **PostgreSQL for message persistence** — every user/assistant message stored; conversation history survives server restarts; fresh SDK session created on restart with prior context injected

---

### Decision 5: SSE Event Abstraction via EventBus ✅

Decouple SDK events from the HTTP response writer. SDK events → EventBus → multiple independent consumers (SSE writer, message persister, metrics collector, future WebSocket transport). Enables testability and future extensibility without touching the SDK integration.

---

### Decision 6: Copilot SDK Session Hooks for Observability ✅

Use SDK's built-in hooks for LLM-level tracing without modifying business logic:
- `onSessionStart` → log session creation (model, mode, userId, projectId)
- `onPreToolUse` → log tool name, args (sanitized), timestamp
- `onPostToolUse` → log result size, latency, success/failure
- `onErrorOccurred` → log errors with context, decide retry/skip/abort
- `onSessionEnd` → log session duration, total message count, total tool calls

### Phase 2 — Resilience & Scale (P1)

#### 2.1 Redis: Session Metadata + Sticky Routing

SDK sessions are not serializable — they hold live JSON-RPC connections to the CLI process. Strategy: in-process `Map` keeps live sessions, Redis keeps metadata to support sticky routing in multi-instance deployments.

**Redis key structure:**
```
session:metadata:{sessionKey}  →  { userId, mode, projectId, instanceId, lastUsedAt, createdAt }  TTL: 35min
instance:sessions:{instanceId} →  SET of sessionKeys hosted on this instance
```

**Files to create/modify:**
- New `apps/server/src/lib/redis.ts` — ioredis client, connection lifecycle
- New `apps/server/src/lib/session-registry.ts` — `register()`, `touch()`, `unregister()`, `getOwnerInstance()`, `isAlive()`
- `apps/server/src/chatAgent.ts` — sync session lifecycle with Redis (on create, destroy, idle)
- `apps/server/src/index.ts` — Redis connection open/close in startup/shutdown
- `apps/server/package.json` — add `ioredis`

**Load balancer config:** sticky sessions by cookie (`SDLC_INSTANCE_ID`) or IP hash.

---

#### 2.2 SSE Event Abstraction Layer (EventBus)

Replace the tight coupling in `routes/chat.ts` (SDK event → `res.write()` directly) with a typed EventBus.

```
SDK events
    │
    ▼
EventBus (typed Node.js EventEmitter)
    ├── SSE Writer        → res.write() per event type
    ├── Message Persister → messageStore.insertMessage()
    ├── Metrics Collector → increment counters, record latency
    └── Audit Logger      → structured log with user + content reference
```

**Files to create/modify:**
- New `apps/server/src/lib/event-bus.ts` — typed EventEmitter, maps SDK event types to normalized internal event types
- New `apps/server/src/lib/event-handlers/sse-writer.ts`
- New `apps/server/src/lib/event-handlers/message-persister.ts`
- New `apps/server/src/lib/event-handlers/metrics-collector.ts`
- `apps/server/src/routes/chat.ts` — replace per-event `res.write()` calls with EventBus emissions; SSE writer subscribes to bus; cleanup still via `res.on("close")`

### Phase 4 — Polish (P3)

#### 4.1 Error Handling & Resilience
- **Circuit breaker** for GitHub API calls — `opossum` library; fail fast when GitHub API is down; fallback to LRU cache if possible
- **SDK process crash recovery** — catch `session.send()` failures caused by dead CLI process; attempt session recreation once before surfacing error to user
- **Graceful degradation UX** — if KB unavailable, show dismissible banner but allow general chat

#### 4.2 Security Hardening
- **Prompt injection defense** — detect common injection patterns in user input before sending to SDK; log suspicious inputs
- **Audit log table:**
  ```sql
  CREATE TABLE audit_log (
    id         BIGSERIAL PRIMARY KEY,
    user_id    TEXT,
    action     TEXT NOT NULL,
    resource   TEXT,
    details    JSONB,
    ip         INET,
    created_at TIMESTAMPTZ DEFAULT NOW()
  );
  ```
- **Per-tenant rate limiting** — current rate limiter is global; add tenant-level quotas
- **Content Security Policy** — tighten Helmet CSP headers for Next.js frontend

#### 4.3 Frontend Improvements
- Conversation sidebar — list past sessions by project, click to reload history
- Markdown editor for UD review/editing before approval
- Export options — PDF (client-side `jspdf`), Confluence push, Jira ticket creation
- Keyboard shortcuts — Ctrl+Enter to send, Ctrl+N for new chat

## Part 6 — KB Quality Checklist (Ongoing)

For the Agentic Workflow that generates KB files on PR merge, ensure:

- [ ] `INDEX.md` entries include a 1-line description of each file's content (not just the filename)
- [ ] `INDEX.md` is organized by module/domain with clear section headers
- [ ] Each module doc includes: purpose, key classes/functions, data models, integration points
- [ ] Docs reference other related files explicitly (helps LLM chain fetches)
- [ ] No single KB file exceeds 30,000 characters (add size check to generator)
- [ ] `README.md` gives a high-level project map so LLM can orient itself on first message
